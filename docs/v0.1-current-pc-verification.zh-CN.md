## 4. Provider 实测

测试只输出状态和能力布尔值，不记录 Key 或回答正文。`32768` 是低成本参数接受测试，不是官方精确最大输出。

| Provider | 模型列表 | 短文本 | 接受 32768 | 函数调用 | Reasoning |
| --- | --- | --- | --- | --- | --- |
| OpenCode Go / `glm-5.2` | 200，20 个模型 | 通过 | 是 | 通过 | `thinking:{type:"enabled"}` 通过 |
| 已保存自定义中转 / `claude-opus-4-8` | 200，4 个模型 | 通过 | 是 | 通过 | 未在未知 relay 协议上强制注入 |
| 内置中转 / `gpt-5.5` | 502 | 502 | 无法判断 | 无法判断 | 无法判断 |
| DeepSeek 官方 / `deepseek-v4-pro` | `/models` 未返回可测试模型 | 流式与非流式均 200 | 未探测 | 未探测 | `auto` 转 `adaptive` 后通过 |
| MiniMax 中国区官方 | 能力探测 | 基础对话可用 | 未探测 | 未探测 | 未完整探测 |

内置中转的 502 表示上游在验证时不可用，不能据此判断模型输出或工具上限。两个可用 OpenAI-compatible `/models` 响应都没有给出 `max_tokens`、`max_output_tokens` 或 `context_window`，因此 Bridge 不伪造精确上限。

## 6. 验收候选产物（发布前）

```text
dist\candidate-deepseek-thinking-fix-20260711\claude-science-assistant-v0.1.3-release-portable.zip
dist\candidate-deepseek-thinking-fix-20260711\claude-science-assistant-v0.1.3-release-portable.zip.sha256
```

最终发布时，ZIP 的权威 SHA256 以同目录、同名的 `.zip.sha256` 文件为准。不要把 ZIP
自身的哈希写死进会被打包回 ZIP 的文档，否则文档变化会再次改变 ZIP 哈希。

候选包检查结果：

- EXE 哈希与 `manifest.json` 一致。
- `vendor/claude-science/linux-x64/claude-science` 存在且与 vendor manifest 哈希一致。
- 包含能力探测脚本、公开文档、Skill 和运行时脚本。
- 不包含 `private/`、嵌套 `.rar/.zip/.7z`、用户设置或疑似 `sk-...` Key。
- `manifest.json` 记录 `sourceTreeDirty=true`，因为本地修改尚未提交；发布前必须先完成有意提交并重新构建最终资产。

## 7. 尚未完成的发布门槛

1. 复核并提交当前工作区，只显式暂存公开源码/文档，不使用无差别 `git add .`。
2. 提交后重新构建 Release ZIP，使 manifest 记录干净且可追溯的源码 commit。
3. 在 309 或另一台不同环境电脑上执行完整升级/安装验收，重点检查 Ubuntu-22.04、旧目录接管、Key 切换、磁盘告警和 rollback PlanOnly。
4. 复测内置中转；502 恢复前不能宣称其文本、工具或 reasoning 能力正常。
5. 用户审批发布计划后，再推送分支/合并 main、创建 v0.1.3 Release，并核对远端 ZIP 与 `.sha256` 资产完整性。

当前结论：**v0.1.3 本机发布候选已经可运行并通过本地完整验收，但尚未达到“已公开发布”状态。**
