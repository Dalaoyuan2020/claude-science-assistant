# CSA v0.2 安装、升级、打包与回退

## 1. 统一原则

- 发布物是完整 ZIP，不是单独 EXE。
- 新包解压到新目录，不覆盖正在使用的旧目录。
- 先做只读预检，再安装/修复运行时，再启动服务。
- 升级成功前保留旧包；失败时退出新启动器并重新打开旧包。
- 不迁移、不注销、不重建现有 WSL 发行版。

## 2. 全新安装

1. 下载 ZIP 和同名 `.sha256`。
2. 校验 SHA-256 后完整解压到短路径，例如 `D:\Apps\CSA-v0.2.0`。
3. 运行 `1-run-acceptance-preview.bat`。它只读检查文件、WSL 和运行时条件。
4. 已有可用 Ubuntu 时运行 `4-install-runtime-after-preview.bat`。
5. 没有 WSL 时，以管理员 PowerShell 显式执行安装流程；安装 Windows 功能和 Ubuntu 可能需要重启。
6. 打开 `3-open-claude-science-assistant.bat`，添加 Provider/API Key。
7. 打开 Connect，配置 Telegram 或飞书；安装浏览器插件时在 Chrome 扩展开发者模式选择包内 `extensions/csa-claude-science-connector`。
8. 运行 `scripts/verify-v0.2-package.ps1` 做包内基础验收；服务已启动时可加 `-LiveConnectRuntime -VerifyProxy`。

API Key 使用 Windows 当前用户 DPAPI，不能从另一台电脑或另一 Windows 用户复制。换机后必须重新添加。

## 3. 从 v0.1.x 并排升级

1. 保留旧解压目录和旧 ZIP。
2. 退出旧 CSA 启动器；不要在升级期间进行 WSL 迁移实验。
3. 将 v0.2 ZIP 解压到新的独立目录。
4. 从新目录运行只读预检，再运行安装/修复。脚本会更新项目管理的 Bridge、Claude Science 二进制和两个 Skill，不会删除用户的 WSL 数据。
5. 打开新启动器并确认 Bridge 来源路径指向新目录。
6. 同一 Windows 用户、同一电脑上的 DPAPI Provider 列表可继续使用；Connect 的 SQLite 和 WSL 配置位于用户数据目录，可继续使用。
7. Chrome 开发者模式插件如果仍指向旧包目录，先移除或点击“重新加载”，再选择新包的 extension 目录。
8. 完成 Telegram 文本、图片和 Subagent Demo 验收后，才删除旧解压目录。

不要只替换 EXE。Gateway、Skill、Bridge、浏览器插件和文档必须保持同一版本。

## 4. 数据兼容

| 数据 | 位置 | 升级策略 |
|---|---|---|
| Provider 凭据 | Windows 当前用户 DPAPI | 同机同用户复用；跨机重录 |
| Bridge 配置 | WSL 用户数据目录 | 原位保留，启动时校验 source path |
| Connect SQLite | `~/.local/share/claude-science-api-bridge/connect/connect.db` | 原位打开，迁移必须向前兼容 |
| Connect Token/配置 | WSL 用户数据目录 + DPAPI | 不写入发布包 |
| Subagent 请求与结果 | 项目 `reports/` | 保留；新增 outbox 无需迁移旧 run |
| 浏览器插件 token | Chrome `storage.local` | 新目录重载后可重新一键连接 |

更新前可退出 Gateway 后复制 `connect.db` 作为本地备份。不要在 Gateway 运行时手工编辑 SQLite。

## 5. 回退

1. 从托盘明确退出新启动器。
2. 确认新 Gateway 和 Bridge 已停止。
3. 打开旧包的启动器。
4. 让旧包重新接管 Bridge，并运行其 `status-probe.ps1`。
5. 保留 v0.2 创建的数据库和 reports；不要通过删除数据库来“修复”升级。

如果旧版本无法读取新 Connect 数据库，先恢复升级前备份；不要降级 schema 或直接改表。

## 6. 构建正式包

在仓库根目录执行：

```powershell
$env:CARGO_BUILD_JOBS = "1"
.\scripts\verify-new-features.ps1 -LiveExternalAgent -LiveConnectRuntime -VerifyProxy
.\scripts\package-launcher-portable.ps1 -Profile release
```

生成：

```text
dist/claude-science-assistant-v0.2.0-release-portable.zip
dist/claude-science-assistant-v0.2.0-release-portable.zip.sha256
```

GitHub 公开资产必须从干净提交重建，并满足：

```text
manifest.version = 0.2.0
manifest.packageQualifier = 空
manifest.sourceCommit = 发布提交
manifest.sourceTreeDirty = false
```

## 7. GitHub 发布流程

1. 审核 `git diff`，只纳入本版本功能与文档。
2. 在 `codex/` 前缀分支提交，推送并创建 PR。
3. CI/本机测试通过后合并到 `main`。
4. 从最终干净 `main` 提交重建 ZIP，不复用脏工作区候选包。
5. 创建 `v0.2.0` 注释标签和 GitHub 草稿 Release。
6. 上传 ZIP 与 `.sha256`，使用 `docs/github-release-v0.2.0.md` 作为说明。
7. 回下载两个资产，重新校验 SHA-256，并从新目录执行包内验收。
8. 回下载验收通过后公开 `v0.2.0` 正式 Release，并保留不可变资产与哈希。

发布期间不应使用 `git add -A`；必须精确审查提交范围，并从最终干净 `main` 构建资产。
