# CSA v0.2 技术实现与发布审查

## 1. 产品边界

CSA 桌面端是本机能力与审批中枢，不重做完整聊天客户端。它负责：

- 启停 WSL Bridge 与 Claude Science。
- 管理 API Key、运行时状态和只读诊断。
- 接收飞书/Telegram 消息并路由到当前科研工作区。
- 在沙盒遇到能力边界时，人工批准本机外部 Agent 会话。

浏览器插件、Connect Skill 和 External Agent Skill 都是适配层，不拥有宿主机任意命令权限。

## 2. Connect 技术栈

| 层 | 技术 | 职责 |
|---|---|---|
| 桌面 UI | React 19、TypeScript、Vite | 状态、配对、项目绑定、队列和只读历史 |
| 本机外壳 | Tauri 2、Rust | DPAPI、进程生命周期、插件本地端点、UIAutomation 回退 |
| 消息网关 | Go 单文件 Linux 二进制 | Telegram/飞书、SQLite、队列、状态机、MCP、媒体发送 |
| 持久化 | SQLite | 游标、身份、绑定、消息、投递状态、审计事件 |
| Telegram | Bot API `getUpdates` 长轮询 | 不开放公网端口的双向私聊 |
| 飞书 | 官方 Go SDK 长连接 | `im.message.receive_v1` 与回复接口 |
| Claude Science 连接 | MCP Streamable HTTP | 领取消息、回写进度与回复 |
| 页面投递 | Chrome Manifest V3 | 页面内 DOM 定位、提交确认、回复观察 |
| 回退投递 | Windows UIAutomation、工作区文件 | 插件不可用或 MCP 不可用时降级 |

Gateway 只绑定 WSL `127.0.0.1:9881`，MCP 使用高熵 Bearer Token。聊天平台凭据由 Windows 当前用户 DPAPI 保存，传入 WSL 时不进入命令参数和日志。

### 消息状态与幂等

```text
received -> authorized -> bound -> queued -> claimed
-> submitted -> replied | needs_local_approval | failed | expired
```

平台事件 ID 与本地消息 ID 共同用于去重。通道游标和消息状态持久化在 SQLite，重启后恢复。无法确认提交结果时进入 `delivery_unknown` 并停止自动重试，避免同一消息二次投递。

### 半流式回复

回复按阶段更新而非逐 token 转发：已接收、处理中、阶段结果、最终结果。Gateway 对进度更新做节流和合并，最终回复仍只有一条权威完成记录。

## 3. Subagent 技术栈

| 层 | 技术 | 职责 |
|---|---|---|
| 沙盒入口 | `csa-external-agent` Skill + Bash/Python | 生成脱敏、人工审批请求 |
| 交换格式 | JSON 文件、原子改名 | inbox/outbox 稳定接口 |
| 任务管理 | Tauri/Rust | 扫描、校验、批准、run/session/outbox 管理 |
| 执行器 | 本机 Claude Code CLI | headed、plan permission mode、可恢复 session |
| 会话记录 | Claude Code JSONL + CSA run 目录 | 只读同步历史与结果 |

关键目录：

```text
reports/csa-agent-inbox/<requestId>.json
reports/csa-agent-runs/<runId>/
reports/csa-agent-outbox/<requestId>.json
```

默认策略为 `manual-only`。CSA 打开的外部会话使用 plan 权限模式；任务正文只作为 Agent 输入，不作为 shell 命令。request ID、run ID 和 resultPath 都经过路径边界检查。

## 4. 浏览器插件的价值

插件不是第二个客户端。它只提供：

- 确认当前标签页属于本机 Claude Science。
- 精确定位页面 composer。
- 投递文本并验证输入框清空或用户消息出现。
- 上报页面就绪、提交结果与必要的回复摘要。

最终优先级：

```text
Browser Extension DOM
-> Windows UIAutomation
-> Connect queue / MCP / Skill fallback
```

这种分层允许无插件时继续使用，同时降低纯屏幕坐标操作的脆弱性。

## 5. 安全审查

- 不修改 VPN、系统代理、DNS、hosts、证书或 443 端口。
- 外部聊天文字永不解释为 shell。
- 插件只请求 `localhost:8765`、`127.0.0.1:8765` 和本地插件桥端口。
- API Key、Bot Token、App Secret、MCP Token 不进入前端状态、日志、进程参数或证据 JSON。
- 图片只从受管 artifact 根目录只读解析，并校验时间、路径、MIME、魔数、大小和 SHA-256。
- WSL 存储迁移继续是只读诊断与方案，不由启动器自动执行。

## 6. 发布审查结论

`v0.2.0` 正式发布为 **GO**。用户已确认 Telegram 文本与图片链路，并授权合并和发布；正式资产仍必须满足：

1. 从干净提交重新构建，`manifest.json.sourceTreeDirty=false`。
2. 从生成 ZIP 解压到新目录做包内验收，不使用源码目录兜底。
3. 回下载 GitHub Release 资产并复核 SHA-256。

飞书需在具体企业租户内独立完成权限配置与验收，不应表述为已在所有企业环境验证。
