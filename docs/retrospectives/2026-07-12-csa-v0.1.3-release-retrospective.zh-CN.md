# CSA v0.1.3 开发、验收与发布复盘

日期：2026-07-12

项目：CSA — Claude Science Assistant

仓库：[Dalaoyuan2020/claude-science-assistant](https://github.com/Dalaoyuan2020/claude-science-assistant)

Release：[CSA v0.1.3](https://github.com/Dalaoyuan2020/claude-science-assistant/releases/tag/v0.1.3)

本文是面向后续文章写作的工程复盘，不是安装教程。它记录这次版本真正解决了什么、哪些判断曾经出错、怎样证明发布包完整，以及哪些能力仍不能提前宣传。

## 一、这次版本真正要解决的问题

CSA 的目标不是再造一个模型平台，而是把 Windows 用户使用 Claude Science 时最容易失败的几层收口起来：

1. Windows 启动器是否能快速、稳定地响应。
2. WSL、Ubuntu、Bridge 和 Claude Science 是否属于同一套运行实例。
3. API Key 切换后，实际请求是否真的切换到新 Provider。
4. 国产模型和中转平台是否能正确接收 Claude 风格的模型名、输出预算、thinking 和工具调用参数。
5. 用户升级时是否必须重装环境，或者可以保留设置和运行时进行安全接管。
6. 大型实验是否可能把 WSL 所在宿主盘写满，并把问题误诊为模型或 Bridge 故障。

因此，v0.1.3 不是单点修复，而是围绕“状态可信、请求可信、升级可回退、发布可验证”做的一次系统收口。

## 二、四项核心改动

### 1. 从五个状态块改成 2×3 六状态体检

首页现在固定展示六项状态：

| 第一行 | 第二行 |
| --- | --- |
| WSL2、运行时、Bridge | Claude Science、WSL 存储、当前 API Key |

这次新增的关键不是“多一个卡片”，而是把 WSL 存储从隐藏基础设施变成产品状态：

- 定位发行版 VHDX 在 Windows 上的真实目录。
- 显示 VHDX 所在宿主盘余量。
- 显示 Linux 根分区余量和 inode。
- 显示 VHDX 大小。
- 检测根文件系统、`/tmp` 和用户目录是否可写。
- 达到危险阈值时阻止盲目重启，避免文件系统问题继续扩大。
- Bridge 日志超过 50 MB 时只轮转自身日志并保留一份备份。

这项能力来自一次真实故障：大型实验把系统盘或 WSL 宿主盘写满后，表面症状可能是 WSL 报错、Bridge 健康失败、Claude Science 无法运行。若只围绕进程和端口修复，很容易把存储故障当成软件逻辑故障。

必须如实说明边界：v0.1.3 r2 已实现位置检测、C 盘专用预警、容量/可写性告警、危险操作阻断，以及只读“辅助迁移/生成本机 Prompt”按钮。启动器只展示证据并把 Prompt 交给 Codex，不会自动移动 VHDX；仓库已有严格迁移方案和独立审查，实际迁移 Build Gate 仍为 NO-GO。

### 2. 不再把模型输出预算写死，并增强工具调用适配

此前连接测试或协议转换如果统一写入很小的输出值，会造成三类误判：

- reasoning 模型只来得及输出推理块，没有最终文本，被误判为 Key 不可用。
- 模型因长度截断表现得像“主动性下降”。
- 不同平台对 `max_tokens`、`max_completion_tokens` 的要求不同，固定字段会直接触发 400。

v0.1.3 的处理原则是：

- 正式请求保留调用方的输出预算。
- 调用方未提供预算时，不全局注入固定值，让上游采用自身默认。
- 只有显式模型上限配置或上游明确返回限制时才夹取。
- OpenAI o-series 使用 `max_completion_tokens`。
- 连接测试先用 256 tokens；只有同一模型因 reasoning/长度截断时，才受限重试到 1024。
- 明确的 400/422 参数错误最多兼容重试一次；认证、额度、网络和 5xx 不参与伪装重试。

工具调用方面，Bridge 保留并转换：

- Anthropic `tool_choice`。
- 强制函数选择。
- `disable_parallel_tool_use` 到 OpenAI-compatible `parallel_tool_calls`。
- tool schema 根类型约束。
- assistant tool call 与 tool result 的相邻关系。
- 流式文本块和工具块的正确结束顺序。

这类协议细节决定了“模型能聊天”和“模型能作为科研 Agent 工作”之间的差距。

### 3. DeepSeek、MiniMax 和更多 Provider 的兼容

本版继续保持“Provider 是添加入口，不是首页平铺面板”的产品逻辑：

- 官方直连：GLM-5.2、LongCat、DeepSeek、MiniMax、Claude、OpenAI/GPT。
- 聚合平台：OpenCode Go、OpenRouter。
- 中转：项目方自建中转和自定义中转。

具体适配包括：

- MiniMax 使用中国区 Anthropic-compatible 地址。
- DeepSeek 使用官方 Anthropic-compatible 地址。
- DeepSeek/MiniMax 不接受 Claude Science 的 `thinking.type=auto` 时，Bridge 转为 `adaptive` 并移除冲突预算。
- 显式 `enabled` / `disabled` 保持不变。
- 初始模型保持为空，不再钉死 GLM 或其他默认模型。
- 优先读取真实 `/models`。
- DeepSeek 列表不可用时，只尝试经过核验的官方候选，而且必须真实短对话成功后才能保存。
- 单模型映射全部 Claude 角色；多模型按主力/快速意图评分。
- 第三方中转不套用官方候选回退。

验收边界也必须保留：DeepSeek 非流式与流式真实请求已通过；MiniMax 已确认中国区基础连通，但其大输出、函数调用和 reasoning 全能力仍需单独实测，不能用 DeepSeek 的通过替代。

### 4. 从“重装”改成可回退的并排增量升级

CSA 当前是便携包，因此增量升级不是“只下载一个新 EXE”，而是：

1. 下载完整新 ZIP。
2. 解压到新的版本目录。
3. 由新版识别 9876 是否被旧包 Bridge 占用。
4. 只停止能够证明属于 CSA 的旧 Bridge。
5. 从新目录启动 Bridge，并校验 `/health` 的 `source_path` 和配置版本。
6. 复用同一 Windows 用户的 APPDATA 设置和 DPAPI Key。
7. 完成六状态、API 连通和真实对话验收。
8. 验收成功前保留旧目录，失败时可回退。

这使用户无需卸载 WSL、Ubuntu 或 Claude Science 运行时，也不需要每次重填 Key。

它仍不是应用内静默更新器。自动下载、签名验证、设置 schema 迁移、一键回滚和更新通道需要另一个完整设计，不能用“并排升级”偷换概念。

## 三、Bridge 故障带来的关键经验

### 健康不等于归属正确

9876 返回 HTTP 200，只能证明有一个 Bridge 在响应，不能证明它属于当前解压目录。旧包 Bridge 可能健康，但仍在使用旧代码或旧配置。

因此 v0.1.3 把以下证据组合起来：

- 端口是否监听。
- Bridge `/health` 是否响应。
- `source_path` 是否属于当前包。
- 配置 revision 是否与本次切换一致。
- Claude Science 端口是否存在。
- systemd unit 或 fallback 进程是否指向当前目录。

### 源码修复不等于 EXE 已修复

309 电脑上的典型问题是：源码中的 Bridge 检测逻辑已经修正，但正在运行的 EXE 仍由旧提交编译。最终解决方案不是继续修改源码，而是重新编译 release EXE，并核对实际运行文件路径和哈希。

这个经验可以概括成一句话：验收对象必须是最终二进制和最终 ZIP，而不是工作区源码。

### 状态探针和 GUI 必须使用同一个项目根

本机复查时，命令行 `status-probe.ps1` 报告来源匹配，而 GUI 曾提示旧包占用。r2 的 expected/actual 诊断证明两者只有 `/mnt/c` Windows 路径大小写不同。修复策略是仅对 DrvFs 路径做大小写不敏感比较，Linux 原生路径继续严格比较；release EXE 也不再静默回退到编译期源码根。

## 四、发布包为什么不能只上传 EXE

CSA 的运行单元不是单个 Windows 程序，而是一个完整包：

- Windows Tauri EXE。
- `proxy.py` Bridge。
- PowerShell/Bash 启动、体检、验收和回滚脚本。
- `bootstrap-claude-science-wsl` Skill。
- 静态配置面板和 Python 测试。
- 锁定版 Claude Science Linux 二进制。
- 新手文档、升级 Prompt 和 Provider 矩阵。

源码仓库不能提交 vendor Linux 二进制，但 GitHub Release ZIP 必须包含它。这也是早期“仓库有代码、用户下载后却缺运行时”的根因。

v0.1.3 发布时执行了以下闭环：

1. 从干净 Git 提交构建，manifest 记录 `sourceTreeDirty=false`。
2. 检查 EXE 哈希与 manifest 一致。
3. 检查 Claude Science Linux 二进制与 vendor manifest 一致。
4. 扫描 ZIP，拒绝 `private/`、嵌套压缩包和疑似 Key。
5. 同时上传 ZIP 与 `.sha256`。
6. 从 GitHub 重新下载远端资产。
7. 再次比较大小、SHA256、source commit 和 vendor 二进制。

首发 ZIP（r2 使用独立文件名和新 `.sha256`，避免静默替换）：

```text
claude-science-assistant-v0.1.3-release-portable.zip
SHA256 e0e92e53dcfdb86056b9dfb6b2635cc49e14ea5f03f4c1ea88f9f2f608545b78
```

## 六、本次发布暴露出的流程问题

### 1. 文档会落后于代码

代码已经变成六状态布局，但 README 仍引用旧版五状态截图；更新记录也仍写着“尚未发布”。如果只看测试通过，很容易把这种发布质量问题漏掉。

改进：发布后必须重新读取远端 README、Release body 和资产列表，而不是只验证 Git push 成功。

### 2. 计划、实现和宣传必须分层

存储迁移已有详细计划和外部审查，并不等于功能已上线。文档必须使用三种明确状态：

- 已实现并验证。
- 已设计但 Build NO-GO。
- 后续建议，尚未进入版本。

### 3. “真实请求成功”不能替代完整 Provider 能力矩阵

一个短文本请求成功，只能证明基础连通。输出上限、工具调用、图片、reasoning 和流式事件都要分别验证。

### 4. GitHub Contributors 属于历史和缓存问题

Contributor 不是 README 文案。必须检查默认分支历史、GitHub commit author 关联和统计 API。当前远端默认分支已只包含项目作者，页面残留来自旧统计缓存；在没有目标提交可改写时，不应为了“看起来做了事”而强推历史或重发资产。

### 5. 自测脚本也必须能从半失败状态恢复

包内 `.venv` 第一次安装依赖失败后仍会保留 Python 可执行文件。旧 self-test 只判断 Python 是否存在，因此第二次运行会跳过依赖安装并报模块缺失。r2 改为先执行依赖导入检查，缺失时再修复 requirements，使“网络镜像临时失败后直接重跑”成为可验证路径。

## 十、结论

v0.1.3 的最大价值不是“又支持了几个模型”，而是建立了三个可信闭环：

1. 状态闭环：从进程、端口、健康、来源、配置和存储共同判断系统是否可用。
2. 请求闭环：从 API Key 测试、模型发现、参数转换到真实流式/非流式请求验证。
3. 发布闭环：从干净提交、完整 ZIP、二进制哈希到 GitHub 远端重新下载复核。

真正稳健的软件，不是承诺永远不出错，而是让用户知道错误发生在哪一层、下一步是否安全，以及如何回到上一个可用状态。
